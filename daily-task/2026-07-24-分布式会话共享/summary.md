# 线上踩坑点与拓展方案

## 一、真上生产才会碰到的坑

### Redis 会话相关
1. **Redis 变成新的单点**：会话全存 Redis，Redis 一挂全站掉登录。务必做**主从 + 哨兵**或**集群**，别用单实例。
2. **序列化坑**：Spring Session 默认把对象用 JDK 序列化存进 Redis。一旦用户类改了字段，老数据反序列化可能报错。建议改成 **JSON 序列化**，兼容性更好、也方便排查（能直接看懂存了啥）。
3. **过期时间与"记住我"打架**：TTL 设太短用户老掉线，设太长又占内存、也不安全。常见做法是**活跃就续期**（每次访问刷新 TTL），配合"记住我"用更长的独立令牌。
4. **Cookie 没设对**：sessionId 放 Cookie 时要加 `HttpOnly`（防 JS 偷取）、`Secure`（只走 HTTPS）、`SameSite`（防 CSRF）。漏了就是安全漏洞。
5. **缓存穿透式风暴**：大量请求带着无效 sessionId 疯狂查 Redis。可在应用层做**空结果短缓存**或限流。

### JWT 相关
1. **想踢人下线却踢不掉**：JWT 签发后到期前一直有效。要支持"立即失效"，得引入 **Redis 黑名单**（把要作废的 token 记进去，验签后再查一次黑名单）——这等于又加回了存储，要提前想清楚是否值得。
2. **令牌里塞敏感信息**：Payload 只是 Base64 编码、**不是加密**，任何人都能解开看内容。绝不能放密码、身份证等敏感数据。
3. **密钥泄露 = 灾难**：谁拿到密钥就能伪造任意用户的令牌。密钥要够长够随机，并放在配置中心/环境变量里，别硬编码进代码提交到 Git。
4. **令牌太大**：信息塞太多会让每个请求头变胖，浪费带宽。只放必要的 userId、角色即可。
5. **时钟不同步**：多机校验 `exp`（过期时间）时，各机器系统时间要用 NTP 对齐，否则可能出现"这台说过期了、那台说没过期"。

### IP Hash 相关
1. **负载不均**：大客户或同一出口 IP 的一群人（如整个公司走同一个网关 IP）会被压到同一台机器。
2. **扩缩容会打乱分配**：机器数一变，取模结果全变，大批用户会话瞬间失效。想缓解可用**一致性哈希**，但那已经比直接上 Redis 复杂了，不如直接换方案。

## 二、同类场景拓展

- **单点登录 SSO**：多个子系统（如企业内的邮箱、OA、报销）共享一次登录。本质仍是"会话集中管理 + 令牌传递"，Redis 会话或 JWT 都是其底层积木。
- **网关统一鉴权**：微服务架构下，把验签/查会话统一放在**API 网关**做，后端各服务只管业务、拿网关透传的用户身份即可。这样鉴权逻辑只写一处。
- **多端登录与互踢**：手机、网页、Pad 同时登录一个账号的管理。用 Redis 存"userId -> 当前有效 sessionId 列表"，想只保留最新一个（互踢）还是允许多端并存，都能灵活控制——这正是 JWT 难做、Redis 会话好做的地方。
- **灰度/滚动发布**：正因为做到了无状态，发布新版本时才能一台台平滑替换，用户完全无感，不会因为"重启机器丢会话"而被迫重新登录。

## 三、一句话总结

**把会话状态从机器里剥离出来，是水平扩展的前提。** 普通业务优先 Redis 集中存储（可控、能踢人）；对外 API、权限稳定则可用 JWT（无状态、省查询）；IP Hash 只当临时救急。选型的核心永远是那道选择题：**你更需要"能随时管控"，还是"极致无状态"？**

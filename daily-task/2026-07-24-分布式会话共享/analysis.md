# 拆解：多机部署下如何共享登录状态

## 一、这道题到底在考什么

表面问"怎么共享登录信息"，背后其实是在考你对**"有状态"和"无状态"**的理解。

- **有状态**：服务器自己记着"你是谁"（Session 存在本机内存）。好处是快，坏处是这份记忆只有它自己有，换台机器就抓瞎。
- **无状态**：服务器自己啥也不记，"你是谁"这个信息要么放在大家共用的公共仓库（Redis），要么由客户端自己带着（JWT）。好处是任何一台机器都能处理任何请求，坏处是多一次查询或验签的开销。

多机部署要解决的，就是把"有状态"改造成"无状态"或"共享状态"。这是分布式系统入门的经典一课。

## 二、业务痛点：为什么必须解决

| 痛点 | 通俗解释 |
| --- | --- |
| 登录了却提示未登录 | 请求被负载均衡随机分到没有你档案的机器 |
| 用户体验崩坏 | 用户以为系统有 bug，频繁掉登录，投诉率飙升 |
| 无法水平扩容 | 如果靠"绑定机器"续命，就没法自由加减机器抗流量 |
| 单点故障放大 | 会话只在一台机内存里，这台机一挂，一批人全掉线 |

一句话：**会话状态和具体机器绑死，是水平扩展的最大绊脚石。**

## 三、三种方案逐一讲清"怎么实现"

### 方案一：Redis 分布式会话（推荐，生产主流）

**核心机制**：把 Session 从"每台机的本地内存"搬到"所有机共用的 Redis"。

调用链（数据怎么流动）：

```
浏览器 --(带 sessionId 的 Cookie)--> 负载均衡 --(随机)--> 任意一台服务器
                                                          |
                                     拿 sessionId 当 key 去 Redis 取/存档案
                                                          |
                                                        Redis（公共储物柜）
```

关键数据结构（存在 Redis 里的样子）：

- key：`session:{sessionId}`，例如 `session:a1b2c3d4...`
- value：一个哈希/JSON，装着 `userId`、`userName`、`roles(权限)`、`loginTime` 等
- 额外设置**过期时间（TTL）**，比如 30 分钟。用户一直活跃就续期，长时间不动自动清理——这就是"配合过期策略自动清理无效会话"。

登录与校验的两步：

1. **登录时（写）**：校验账号密码 → 生成随机 sessionId → 把用户档案写进 Redis 并设 TTL → 通过 Cookie 把 sessionId 发回浏览器。
2. **后续请求（读）**：从 Cookie 取出 sessionId → 去 Redis 查 → 查到就是已登录（顺手续期 TTL），查不到就是未登录/已过期。

技术选型与权衡：
- 现成方案 `Spring Session + Redis`，几行配置就能把原来的本地 Session 透明地换成 Redis 存储，业务代码几乎不用改。
- 优点：应用彻底无状态，可随意加减机器；踢人下线只需删 Redis 里对应 key，非常方便。
- 代价：每次请求多一次 Redis 网络查询（但 Redis 极快，通常零点几毫秒）；Redis 本身要做主从/集群保证高可用，否则它就成了新的单点。

### 方案二：JWT 无状态令牌

**核心机制**：不存档案，把信息签进令牌，客户端自己带。

一张 JWT 由三段组成，用点号 `.` 连接：`Header.Payload.Signature`

- **Header（头）**：说明用了哪种签名算法。
- **Payload（载荷）**：装着 userId、用户名、过期时间等（注意：这部分只是 Base64 编码，**不是加密**，别放密码等敏感信息）。
- **Signature（签名）**：用服务端密钥对前两段算出的"防伪章"。改动前两段任意一个字，签名就对不上，服务端立刻识破。

调用链：
```
登录成功 → 服务端用密钥生成 JWT → 发给客户端保存
之后每次请求带上 JWT → 服务端仅验签(重算签名比对) → 通过即识别用户，不查任何存储
```

权衡：
- 优点：服务端零存储、天然适合多机（每台机有同一把密钥就行）、省掉一次查库。
- 缺点：**难以中途撤销**。令牌签发后在过期前一直有效，想立刻踢人下线得额外搞"黑名单"（又要引入存储，等于绕回去了）。所以适合权限变动不频繁的场景。

### 方案三：IP Hash 粘性会话（临时方案）

**核心机制**：负载均衡按客户端 IP 做哈希，让同一 IP 每次固定命中同一台机器，于是 Session 放本机内存就够用。

权衡：
- 优点：改动最小，业务代码一行不动。
- 缺点：负载可能不均（大客户集中在某台）；那台机器宕机，它上面的会话全部丢失；同一出口 IP 的一群人（如同一公司）会被压到一台机上。只适合过渡救急。

## 四、三者怎么选（一张对比表）

| 维度 | Redis 会话 | JWT | IP Hash 粘性 |
| --- | --- | --- | --- |
| 服务端是否存状态 | 存在 Redis（集中） | 不存 | 存在某台机内存 |
| 能否随意扩缩容 | 能 | 能 | 受限 |
| 踢人下线/改权限 | 方便（删/改 key） | 麻烦（需黑名单） | 方便 |
| 单点故障影响 | Redis 需高可用 | 无 | 一台挂掉丢一批会话 |
| 每请求额外开销 | 一次 Redis 查询 | 一次本地验签 | 无 |
| 适用场景 | 生产主流、需可控 | 权限稳定、追求无状态 | 临时过渡 |

## 五、给新手的选型建议

- 大多数需要"能随时踢人、能改权限"的后台系统 → 选 **Redis 会话**，稳妥可控。
- 对外开放 API、追求极致无状态、权限不常变 → 选 **JWT**（或 JWT + Redis 黑名单折中）。
- 只是临时上线救急、来不及改造 → 用 **IP Hash** 顶一下，后面再换。

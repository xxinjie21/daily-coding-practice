## 52. 把单体项目进行了多机部署，多台服务器是如何共享用户登录信息的？

### 一句话题目
一个网站原来只有一台服务器，现在为了扛住更多访问，部署成了好几台。问题来了：用户在 A 机器上登录成功了，可下一次请求被转发到了 B 机器，B 机器不认识他，于是页面提示"你还没登录"。**怎么让所有机器都认得同一个已登录的用户？**

### 用大白话讲清楚问题在哪
先打个比方：

- 「登录状态」就像你去游乐园买了张手环，进门检票时工作人员确认"这人买过票"。
- 单台服务器时代，只有一个检票口，手环信息就记在这个检票口的小本子上，进出都查这个本子，没问题。
- 现在游乐园火了，开了好几个检票口（多台服务器）。你在 1 号口登记了手环，可你想玩过山车时走到了 3 号口，3 号口的小本子上根本没有你的记录 —— 它当然说"没见过你，重新买票吧"。

技术上，用户登录后服务端会生成一份**会话状态（Session）**。所谓 Session，通俗说就是服务器给你单独建的一个"小档案袋"，里面装着"你是谁、你有什么权限"。单机时这个档案袋放在这台机器自己的内存里；多机部署后，请求可能被负载均衡（一个在多台服务器前面分流量的"大堂经理"）随机分给任意一台机器，而档案袋只存在最初那台机器的内存里，别的机器自然找不到。

### 原题给出的三种解决思路（后面就照这三种来做）

**思路一：Redis 做分布式会话存储（生产主流）**
把"档案袋"从每台机器自己的内存里搬出来，集中放到一个大家都能访问的公共仓库——Redis（一个超快的内存数据库，可以理解为"所有机器共用的一个高速储物柜"）。
- 用户登录后，服务端生成一个 `sessionId`（一串随机字符，相当于储物柜的"钥匙编号"），把用户信息存进 Redis，key 就是这个 sessionId。
- 这个 sessionId 通过 Cookie 发回给浏览器。之后每次请求浏览器都带上它。
- 无论请求落到哪台机器，机器都拿着 sessionId 去 Redis 这个公共储物柜取档案 —— 所有机器看到的都是同一份数据。
- Spring 生态里的 `Spring Session + Redis` 就是这套的现成实现，应用服务器变成"无状态"（自己不存任何登录信息）。

**思路二：JWT，把信息签在令牌里（服务端不存）**
JWT（JSON Web Token）换了个思路：干脆不在服务端存档案，而是把用户信息直接写进一张"防伪令牌"里发给客户端自己保管。
- 令牌里带一个用**服务端密钥**算出来的签名，别人改不了、伪造不了。
- 每次请求客户端把令牌带上，服务端只需**验签**就知道"这是真的、没被篡改"，从而识别用户，全程不查任何存储。
- 缺点：令牌一旦签发，在过期前很难中途作废（想踢人下线比较麻烦），适合对实时权限控制要求不高的场景。

**思路三：负载均衡 IP Hash 粘性会话（临时方案）**
让负载均衡这个"大堂经理"记住：同一个客户端 IP 每次都固定分到同一台机器。这样档案袋放在那台机器内存里就够用了。
- 缺点：破坏了负载均衡的均匀性；而且那台机器一挂，落在它上面的一批用户会话全丢。一般只当临时救急用。

### 结论
真正的生产环境、尤其是微服务架构下，**Redis 集中存储是主流**。它高可用，还能配合"过期时间"自动清理没用的旧会话，脏活累活都甩给中间件干。淘宝、美团这类大流量系统基本都是这么扛住的。

# 分析：几行代码实现一个负载均衡器（大白话版）

> 这篇文章不假设你是大佬。每个技术词第一次出现，我都会用生活例子讲清楚。

## 一、负载均衡到底是啥？
打个比方：你去银行办业务，门口有个**大堂经理**。他看哪个窗口人少，就把你指到哪个窗口。结果就是——大家不用全挤在一个窗口，每个人都办得快。

换成系统里的说法：
- **窗口** = 一台台服务器（机器）
- **办业务的人** = 用户的请求
- **大堂经理** = 负载均衡器

负载均衡器干的事，就是把"办业务的人"合理分给"窗口"，别让一台机器累死、其他闲着。

## 二、考点拆解（用大白话说）

### 1. 轮询（Round-Robin）= 轮流来
就像发牌：第 1 张给 A，第 2 张给 B，第 3 张给 C，第 4 张又回到 A。
代码上就是 `i % 机器数`：i 从 0 一直加，`% 3` 的结果就是 0、1、2、0、1、2……循环选机器。

### 2. 并发安全 = 别发重号
现实里请求是"一堆人同时冲过来"（多线程）。如果用一个普通变量 `int i` 来计数，两个人可能同时读到同一个数，导致两台机器接到同一个请求、甚至下标越界。
解决办法：用 `AtomicInteger`（原子整数）。你可以把它想成**一台不会卡壳的取号机**——不管多少人同时来按，它吐出来的号保证不重复、不跳号。

### 3. 加权轮询 = 能者多劳
机器也有强弱。A 是新款、性能好，B 是老古董。如果还"绝对轮流"，B 会被压垮。
于是给机器加**权重**：A 权重 3、B 权重 1，意思是"A 接 3 个、B 接 1 个"。简单做法：把列表展开成 `[A,A,A,B]` 再轮询。
> 例子：权重 5:3:2 的机器，展开后就是 `[强,强,强,强,强,中,中,中,弱,弱]`，轮流选就实现了"强的多干"。

### 4. 工程扩展 = 机器会坏、会换
真实世界机器会下班（宕机）、会新加。不能把机器列表写死在代码里。
- **节点发现**：用 Nacos / Zookeeper 这类"通讯录"自动告诉负载均衡器"现在有哪些机器在线"。
- **健康检查**：定时 ping 一下机器"你还活着吗？"，死了就暂时别分请求给它。

## 三、具体怎么写（照着做就行）

**Step 1 · 准备机器列表 + 一个取号机**
```java
List<String> servers = Arrays.asList("192.168.0.1", "192.168.0.2", "192.168.0.3");
AtomicInteger idx = new AtomicInteger(0);  // 取号机，从 0 开始
```

**Step 2 · 轮询核心（最重要的两行）**
```java
int i = idx.getAndIncrement();           // 取号机吐一个号，并自动 +1
String s = servers.get(i % servers.size()); // 对机器数量取模，循环选一台
```
- 为什么用 `AtomicInteger` 而不是 `int i++`？因为 `i++` 在多人同时来时会发重号（见考点 2）。
- `i % 3` 的结果永远是 0、1、2 循环，正好对应 3 台机器。

**Step 3 · 加权轮询（展开法，原题方法）**
把 `[A,A,A,B]` 当普通列表轮询即可。缺点是：权重高的机器会**连着被打满**（A,A,A,B）。

**Step 4 · 平滑加权轮询（生产改进）**
为了解决"连着打满"，Nginx 用的是**平滑加权轮询**。打个比方：三个人 A、B、C 权重 5:3:2，不让他们连着干，而是穿插着排：`A,B,A,C,A,B,A……`，谁该干谁干，均匀分散。
算法思路（看懂大意即可）：给每台机器记一个"当前权重"，每轮把"当前权重"最大的选出来，选完给它减掉总权重，下一轮再加回来。这样就不会出现 A 连干 5 次。

**Step 5 · 工程补齐（在最小实现之上）**
1. 机器列表从"写死"改为**订阅注册中心**（Nacos/Zookeeper）。
2. 加**健康检查**：探活失败就摘掉这台机器。
3. 加**重试 + 熔断**：某台机器挂了，自动换一台重试，别让用户请求失败。

## 四、本仓库 Demo 对应说明
`code/Demo.java` 严格按原题方法实现：**轮询（取号机取模）+ 加权轮询（展开列表）**，是个能直接跑的最小例子。生产级的"平滑加权、注册中心、健康检查"属于 Step 4/5 的扩展，思路一样，只是工程细节更多。

# 总结：负载均衡器（轮询 / 加权轮询，大白话版）

## 一、线上踩坑点（都是真实会翻车的地方）

1. **计数器没用原子类，发重号了**
   如果你图省事用 `int i++` 而不是直接用 `AtomicInteger`，当很多请求同时来，两个请求可能拿到同一个号，被分到同一台机器，甚至下标越界报错。
   👉 记住：多线程计数，用 `AtomicInteger`（那台不会卡壳的取号机）。

2. **计数器一直加，会不会爆？**
   `i` 会一直涨，很久以后涨到 `Integer` 最大值会"绕回"负数。不过 Java 里对正数取模（`i % 3`）结果依然正确，不用太慌；想更稳可以对 `i` 做 `& Integer.MAX_VALUE` 清掉符号位。

3. **加权轮询"连发"问题**
   本题 demo 用"展开列表"做加权（A,A,A,B），结果是 A 被**连续**打满 3 次。真实系统更爱用 **Nginx 平滑加权轮询**，把请求穿插开，不让强机器瞬间被压垮。

4. **机器写死在代码里**
   `Arrays.asList(...)` 写死 3 台机器，机器下线了你也不知道，还会把请求发给已宕机的机器。
   👉 真实做法：用 Nacos/Zookeeper 自动下发机器列表 + 定时健康检查把坏机器摘掉。

5. **机器下线瞬间，请求怎么办？**
   机器正在重启，但还有请求在半路上。需要**重试 + 熔断**兜底，别让用户直接看到报错。

## 二、同类场景拓展（一句话 + 例子）

- **随机策略**：`servers.get(随机数)`。像抓阄，简单、分散，但不能保证绝对均匀。
- **最少活跃请求数**：看哪台机器"正在忙的请求最少"就分给它，最贴近真实负载（Dubbo 常用）。好比哪个窗口排队人少就指哪。
- **一致性哈希**：同样的请求 key 永远落到同一台机器。适合"有状态/要命中缓存"的场景；加机器时只影响一小部分 key，不会全乱。
- **响应时间权重**：谁 historically 响应快，就给谁多分。Spring Cloud LoadBalancer 支持。
- **客户端 vs 服务端负载均衡**：Ribbon 是在"调用方自己内部"选机器（省一次网络跳转）；Nginx 是在"大门口"统一分发。实际常组合：大门 Nginx → 内部 Ribbon。
- **服务网格（进阶）**：Istio/Envoy 把负载均衡藏进"边车"，业务代码完全不用管。

## 三、落地实现步骤（从 demo 到生产）

1. 先跑通 `AtomicInteger` 轮询骨架（本 demo 已完成），确认请求能均匀分发。
2. 加权重：先用"展开法"，再换成"平滑加权轮询"（更优雅）。
3. 机器列表从写死 → **订阅注册中心**，机器上下线自动感知。
4. 接**健康检查**：探活失败即摘掉，恢复后再加回。
5. 接**熔断/重试**（如 Resilience4j），保护在途请求和故障机器。
6. 跨服务调用优先用现成组件（Spring Cloud LoadBalancer / Dubbo），别从头造轮子。

# 线上踩坑点 & 同类场景拓展

## 一、线上真实踩过的坑（血泪教训）

### 1. 密钥硬编码在代码/配置里 —— 头号大坑
很多人图省事，把 AES 密钥直接写进 `application.yml` 或代码常量。结果代码一上传 GitHub、或者配置被截图外泄，密钥和密文一起流出，加密等于没加。
- **正确做法**：密钥交给 **KMS**（云厂商密钥管理服务）保管，程序运行时动态申请；或至少放进独立的密钥服务，和业务代码、数据库物理隔离。记住一句话：**密文和密钥绝不能放在同一个能被一起偷走的地方。**

### 2. 用了 AES 的 ECB 模式 —— 看似加密，实则漏底
ECB 模式对相同明文永远加出相同密文。比如所有「男」都加成同一串，攻击者不用解密，光看密文分布规律就能猜出内容。
- **正确做法**：用 **GCM**（本 Demo 用的就是它）或 CBC，配随机 IV，让相同明文每次密文都不同，GCM 还自带防篡改校验。

### 3. 忘了「加密后没法搜索」这件事
上线后产品说「要按手机号模糊查用户」，可字段已经 AES 加密了，`LIKE '%138%'` 完全查不了，只能全表解密扫描，性能爆炸。
- **正确做法**：设计阶段就想好查询需求。需要精确查的，额外存一个「**盲索引**」列（对原文加盐哈希后存，查询时对条件同样哈希再匹配）；需要模糊查的，考虑分段哈希或可搜索加密中间件。**别等上线才发现查不了。**

### 4. 接口脱敏了，日志却在裸奔
返回体打了码，结果 `log.info("param={}", param)` 把整个请求对象（含身份证）打进了 ELK，全公司能搜。这是最隐蔽、最常见的泄露源。
- **正确做法**：AOP 统一拦截入参/出参脱敏；异常堆栈、SQL 日志、第三方调用日志也要覆盖；定期用正则扫日志，发现明文身份证/手机号立刻告警。

### 5. 只在前端加密/脱敏，后端裸接
前端加密看着高级，但用户能看到前端代码、能改、能抓包。前端脱敏更是自欺欺人——原文已经传到浏览器了。
- **正确做法**：安全措施**必须在后端**落地，前端加密只能当「锦上添花」，不能当依赖。

### 6. 哈希不加盐 or 全表用同一个盐
不加盐直接被彩虹表秒破；全表共用一个盐，一旦盐泄露等于没加。
- **正确做法**：**每条数据一个独立随机盐**，和哈希值一起存。对密码这类，进一步用 **BCrypt/PBKDF2/Argon2** 这类「慢哈希」，故意让计算变慢，暴力破解成本极高。

## 二、同类场景拓展方案

### 场景 A：用户密码存储
密码是「只比对不还原」的典型，**绝不能用 AES 可逆加密**（否则拖库就全泄露）。用专门的**慢哈希算法** BCrypt / Argon2，它们内置加盐 + 多轮迭代，Spring Security 里 `BCryptPasswordEncoder` 开箱即用。

### 场景 B：银行卡号 / 支付信息
既要能还原（对账、退款）又要能查（按卡号查订单）。方案：AES-GCM 存密文 + 盲索引列存卡号哈希用于查询 + 返回时脱敏 `6222 **** **** 1234`。合规上还要满足 PCI-DSS 要求。

### 场景 C：数据同步 / 数仓
把业务库数据同步到数仓时，敏感字段要么继续保持密文同步，要么在同步管道里做脱敏/加密，别让数仓成为「明文副本」的泄露新入口。

### 场景 D：内部接口 / 微服务间调用
别以为「内网就安全」。微服务之间也可能被横向渗透。核心敏感字段在服务间传递时也建议加密或脱敏，日志同样要脱敏。

---

## 三、一句话总纲
**加密看需求选可逆(AES+KMS)还是不可逆(盐+哈希)；传输靠 HTTPS 打底；返回和日志一律后端统一脱敏；密钥和密文永远分家。** 全链路每个环节都设卡，才叫真正保护了敏感数据。

# 敏感数据加密传输与存储 —— 大白话拆解

## 一、这题到底在考什么？

面试官问「敏感数据怎么加密传输和存储」，其实是在看你**有没有「全链路安全」的意识**。新手常见的坑是：只想到「存库前加个密就行了」，结果传输时明文裸奔、返回时全量吐出、日志里到处是身份证号。真正的答案是——**从数据进门到出门，每一个经手环节都要设卡**。

打个比方：保护敏感数据就像押运一车金条。你不能只在金库门口装保险柜（存储加密），还得管好：路上用不用装甲车（传输加密）、到站清点时给谁看几根（返回脱敏）、门卫值班记录里要不要写清楚金条编号（日志脱敏）。任何一环偷懒，金条都可能丢。

---

## 二、业务痛点（为什么必须做）

| 不做的后果 | 现实例子 |
|---|---|
| 传输明文 | 用户连了公共 WiFi，黑客在同一网络「中间人监听」，手机号身份证全被抓包 |
| 存库明文 | 数据库被「拖库」（整个被脱走），几百万用户信息直接以明文泄露，公司上新闻 |
| 返回不脱敏 | 前端接口返回完整身份证号，随便打开浏览器 F12 就能看到全部原文 |
| 日志打明文 | 运维、外包、日志平台的人都能在 ELK 里搜到用户身份证，内部泄露防不住 |
| 合规风险 | 《个人信息保护法》《数据安全法》明确要求敏感个人信息要加密，做不到要担法律责任 |

## 三、整体架构：给数据全程「设卡」

按数据流动方向，一共 5 道卡：

```
 用户/前端                后端应用                     数据库
   │                        │                           │
   │  ①HTTPS 加密传输        │                           │
   ├───────────────────────▶│                           │
   │                        │ ②按需解密 + 业务处理        │
   │                        │                           │
   │                        │ ③存库前：AES加密 / 加盐哈希 │
   │                        ├──────────────────────────▶│（存的是密文/哈希）
   │                        │                           │
   │                        │ ④查询：解密还原 或 哈希比对 │
   │                        │◀──────────────────────────┤
   │  ⑥返回前统一脱敏         │ ⑤打日志前脱敏 → ELK        │
   │◀───────────────────────┤                           │
```

下面逐个模块讲「具体怎么做」。

## 四、模块一：传输层 —— HTTPS 打底

### 大白话
HTTP 就像用明信片寄信，邮差、中转站谁都能看内容；HTTPS 相当于把信装进一个「只有收件人才能打开的保险箱」再寄。这个保险箱的技术叫 **TLS**（早期叫 SSL）。

### 具体做法
- 全站强制 `https://`，服务端配置 HTTP 自动 302 跳转到 HTTPS，或直接拒绝明文请求。
- 开启 **HSTS**（`Strict-Transport-Security` 响应头），告诉浏览器「以后这个网站只准用 HTTPS」，防止被降级攻击。
- 证书用正规 CA 签发，别用自签证书糊弄。
- 敏感接口可再加一层「应用层加密」：前端用后端下发的公钥（RSA/SM2）加密请求体，后端用私钥解——即使 HTTPS 被某种方式破了，这层还能兜底。

> 小结：HTTPS 是**底线**，不是加分项。没有它，后面做得再花哨也白搭。

## 五、模块二 & 三：存储层 —— 加密 vs 哈希，选对是关键

这是本题的**核心决策点**。先问自己一句话：**这个字段以后还需要「拿回原文」吗？**

### 情况 A：需要还原 → 用「可逆加密」AES

比如实名认证，监管要求时得能导出真实身份证号。这种用 **AES 对称加密**。

- **什么是「对称加密」**：加密和解密用**同一把钥匙**，就像家门钥匙——同一把既能锁门也能开门。速度快，适合大量数据。
- **AES-GCM 模式**：推荐用 GCM 而不是老旧的 ECB。ECB 的毛病是「相同明文永远加出相同密文」，容易被看出规律；GCM 每次加密都带一个随机 **IV（初始向量）**，同样的身份证号每次加出来的密文都不一样，而且自带「防篡改校验」。
- **密钥怎么管**（重中之重）：**绝对不能把密钥硬编码在代码里或写进配置文件！** 要交给 **KMS**（Key Management Service，密钥管理服务，云厂商都有）。程序启动时向 KMS 申请密钥，用完即弃。这样即使代码泄露，没有 KMS 权限也解不了密。
- 进阶：**信封加密**——KMS 里放一把「主钥匙 KEK」，用它加密出很多「数据钥匙 DEK」，DEK 加密业务数据。换钥匙时只需换少量 DEK，不用重加密全表。

### 情况 B：只需比对，不需还原 → 用「不可逆哈希」SHA-256 + 盐

比如「判断这个手机号是不是注册过」「密码校验」，根本不需要拿回原文，只要能比对「是不是同一个」。这种用**哈希**。

- **什么是哈希**：把任意内容压成一段固定长度的「指纹」，同样的输入永远得到同样的指纹，但**反过来算不回原文**（单向）。
- **为什么要加盐（salt）**：光哈希不安全。黑客手里有「彩虹表」——提前把常见值（比如所有手机号）的哈希都算好存成字典，一查就还原。加盐就是在原文后面拼一段随机字符串再哈希，比如 `SHA-256("13812341234" + "x8Kd9#随机盐")`，这样黑客的现成字典全失效，必须为你的每个盐重新暴力算，成本极高。
- 盐要**每条数据一个、随机生成**，并和哈希一起存。

### 一张表帮你选

| 需求 | 方案 | 能否还原 | 典型字段 |
|---|---|---|---|
| 以后要拿回原文 | AES-GCM 加密 + KMS 管钥 | 能 | 身份证号、银行卡号、真实姓名 |
| 只要能比对是否相同 | SHA-256 + 每条随机盐 | 不能 | 登录密码、用于查重的手机号哈希 |

> ⚠️ 有个经典难题：**加密后的数据怎么模糊搜索？** 因为密文没法 `LIKE '%138%'`。常见解法是额外存一个「**盲索引 / 分词哈希**」列——比如把手机号按段哈希（前3位、后4位分别哈希存一列）用于精确匹配查询，或用支持保序/可搜索加密的中间件。本题不展开，知道有这回事即可。

## 六、模块四：接口返回脱敏 —— 统一由后端打码

### 大白话
「脱敏」就是打码，把 `13812341234` 变成 `138****1234`。**关键原则：由后端统一打码，别信前端。** 因为前端代码用户随便就能看、能改，让前端打码等于没打——原文已经吐到浏览器了。

### 具体做法（就是原题给的那段代码）
在 Spring 项目里，给敏感字段挂一个自定义的 Jackson 序列化器，序列化成 JSON 返回时自动打码：

```java
@JsonSerialize(using = SensitiveDataSerializer.class)
private String phone;
```

序列化器内部就是用正则把中间几位替换成 `*`。不同字段掩码规则不同：
- 手机号：`138****1234`（留前3后4）
- 身份证：`110***********1234`（留前3后4，中间全掩）
- 姓名：`张*` 或 `张**`
- 银行卡：`6222 **** **** 1234`

## 七、模块五：日志脱敏 —— 最容易被忽略的漏点

### 大白话
你接口返回打码了，结果一句 `log.info("收到请求：" + request)` 又把原文原封不动写进日志。日志平台（ELK）里一堆人能看，等于开了后门。

### 具体做法
- 用 **AOP**（面向切面编程，可以理解为「在方法执行前后自动插一段统一逻辑」）拦截入参和出参，把 `idCard`、`phone` 等字段自动脱敏后再落日志。
- 或者自定义日志组件 / Logback 的转换器，对匹配敏感格式（如 11 位数字、18 位身份证）的内容自动打码。
- 异常堆栈、SQL 打印、第三方调用日志也都要覆盖到。

## 八、串起来：完整调用链

以「用户提交实名认证」为例，走一遍全链路：

1. 前端通过 **HTTPS** 把身份证号 POST 给后端（路上加密）。
2. 后端 TLS 层解密拿到明文，进入业务方法。
3. 存库前：用从 **KMS** 取来的密钥做 **AES-GCM 加密**，把密文写入数据库；同时（如需查重）算一个 **加盐哈希** 存到盲索引列。
4. 后续查询：需要展示时用 AES 解密还原；只是判断是否存在时直接比对哈希，不用解密。
5. 打日志前：AOP 把身份证号脱敏成 `110***********1234` 再写 ELK。
6. 返回前端前：Jackson 序列化器把返回体里的字段统一打码。

**每一步都设了卡，任意单点被突破，也不会直接泄露全部原文——这就是「纵深防御」。**

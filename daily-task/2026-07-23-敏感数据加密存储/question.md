## 13. 接口中的敏感数据（如身份证号、手机号）应该如何加密传输和存储？

### 一句话题目
后端接口里会用到身份证号、手机号这类「一旦泄露就出大事」的敏感信息，问你：从用户点提交，到数据落库，再到查询返回——**这一整条路上要怎么保护它们不被偷看、不被篡改、不被拖库泄露？**

### 大白话理解「敏感数据」
想象你去银行办卡，柜台要你的身份证。这张身份证一路上会经过很多人手：
- **在路上**（网络传输）：像你把身份证装进信封寄出去，路上会不会被人拆开偷看？
- **在仓库**（数据库存储）：银行把你的信息存进档案柜，万一档案柜被小偷整个搬走（俗称「拖库」），里面是明晃晃的原文还是一堆看不懂的密文？
- **给你看的时候**（接口返回）：柜台屏幕显示你的手机号，是全显示 `13812341234`，还是打码成 `138****1234`？
- **记流水账时**（打日志）：银行的监控录像/流水记录里，会不会不小心把你的身份证号也录进去了？

**只在前端（网页/App）加密是远远不够的**——因为数据一进后端、一落库、一打日志，处处都可能漏。必须「端到端」每个环节都卡死。

### 原题给出的核心思路（后面就按这几条来实现）
1. **传输层必须走 HTTPS**：这是底线。HTTPS 里的 TLS 加密能防「中间人偷听」，所有带身份证、手机号的请求都得用 `https://`，普通 `http://` 直接禁掉。
2. **数据库里不能存明文**，两种典型做法：
   - **要能还原的**（比如实名认证要拿回真实身份证号）：用 **AES 对称加密** 存密文，密钥交给 KMS（密钥管理服务）统一保管，程序里绝不硬编码密钥。
   - **只需比对、不需还原的**（比如判断「这个身份证是不是注册过」）：存 **SHA-256 + 盐值（salt）** 的哈希，注意加盐防「彩虹表」破解。
3. **接口返回时做脱敏**：手机号显示成 `138****1234`，由**后端**统一掩码，别让前端自己处理。Spring 里可以用 Jackson 的 `@JsonSerialize` 挂一个自定义序列化器自动打码。
4. **日志里也别打明文**：用 AOP 拦住参数，把 `idCard`、`phone` 这类字段自动脱敏后再写进日志系统（如 ELK）。

### 原题给出的关键代码片段（脱敏序列化器）
```java
public class SensitiveDataSerializer extends JsonSerializer<String> {
    @Override
    public void serialize(String value, JsonGenerator gen, SerializerProvider sp) throws IOException {
        if (value == null) return;
        if (value.length() == 11) {
            gen.writeString(value.replaceAll("(\\d{3})\\d{4}(\\d{4})", "$1****$2"));
        } else {
            gen.writeString(value);
        }
    }
}
```

### 完整链路（一步都不能少）
```
HTTPS 传输 → 后端解密 → 业务处理 → 存库前 AES加密/哈希 → 查询时按需解密或比对 → 返回前脱敏 → 日志再过滤
```
只要中间少卡一个环节，就可能泄露。

# 分析：几行代码实现一个负载均衡器

## 考点拆解
1. **负载均衡的本质**：在多个服务节点之间按某种策略分发请求，追求高可用与横向扩展。
2. **轮询（Round-Robin）算法**：最朴素、最常用的策略，核心是一个自增计数器对节点数取模。
3. **并发安全**：计数器在多线程环境下必须原子自增，否则会出现重复选取或越界。
4. **加权轮询（Weighted Round-Robin）**：节点性能不均时，按权重分配流量，避免弱机器被打垮。
5. **工程化扩展**：生产环境还需节点发现（注册中心）、健康检查、故障剔除。

## 业务痛点
- 单节点容量有限，流量集中会拖垮服务；需要把请求摊开到集群。
- 朴素轮询对所有节点"一视同仁"，但机器配置往往不同，需要权重区分。
- 节点会上下线，硬编码 `Arrays.asList(...)` 无法动态感知，需注册中心下发 + 健康探测。
- 无健康检查会导致请求被转发到已宕机的节点，出现大量失败。

## 整体架构方案（按原题给出思路落地）
原题给出的最小实现采用**轮询 + 计数器取模**，并扩展到**加权轮询**（按权重把节点展开成列表再轮询）。生产架构在其之上补齐三块：

1. **节点发现**：Zookeeper / Nacos / Eureka 下发可用节点列表，替代硬编码。
2. **健康检查**：定时探活，自动把故障节点从列表中摘除（对应原题"自动剔除故障实例"）。
3. **策略升级**：
   - Nginx 的**平滑加权轮询**（Smooth Weighted RR），避免权重节点连续被打满；
   - Spring Cloud LoadBalancer 提供响应时间权重、随机、最少活跃请求数等策略；
   - 客户端负载均衡（如 Ribbon）在 JVM 内完成节点选择，省一次网络跳转。

> 本仓库 `code/Demo.java` 严格按原题方法实现：轮询 + 原子计数器 + 加权轮询展开，构成可独立运行的最小demo。
